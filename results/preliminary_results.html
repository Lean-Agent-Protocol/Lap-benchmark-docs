LAP Benchmark v2 - Preliminary Results

Measuring AI Agent Comprehension Across API Documentation Formats

February 2026  |  github.com/Lean-Agent-Protocol/Lap-benchmark-docs

Key Insights

LAP outperforms verbose formats LAP-standard scores 0.926 avg vs pretty's 0.888 - a 4.3% improvement in task accuracy while using 85% fewer tokens
Token reduction without quality loss LAP-lean achieves 89.6% token reduction (219K to 23K tokens) while scoring 0.917 - still beating pretty format
Cost and speed gains LAP-standard is 21% cheaper ($0.34 vs $0.44) and 20% faster (69s vs 86s) per task than pretty format
No-doc baseline validates the benchmark Without any documentation (none tier), agents score only 0.427, confirming the benchmark measures documentation comprehension, not prior knowledge

Methodology

The LAP Benchmark v2 is a protocol-agnostic benchmark designed to measure how effectively AI agents can comprehend and utilize API documentation across different formats and compression levels. This preliminary report presents results from 70 benchmark runs across 7 API specifications.

Benchmark Design

7 API specs across 2 formats (OpenAPI, AsyncAPI), 2 tasks per spec, 5 documentation tiers = 70 runs total

Agent Configuration

Claude Sonnet 4.5 with isolated execution per run (fresh session, neutral filename api_docs.txt, no prior context)

Confounding Controls

  • Neutral filenames (no format leakage)
  • Business-language task descriptions
  • Python-only mandate
  • No library hints provided

Documentation Tiers

Tier Description Purpose
none No documentation provided Prior-knowledge baseline control
pretty Original spec in full verbose format (YAML/JSON) Standard industry format baseline
minified Whitespace-stripped version of pretty Basic compression control
lap-standard LAP-compiled with descriptions preserved LAP format with full semantic content
lap-lean LAP-compiled with descriptions stripped Maximum compression with core structure

Scoring System

Protocol-agnostic scoring with three equally weighted components:

Endpoint Identification (35%)

Measures whether the agent identified the correct API endpoints needed to complete the task

Parameter Accuracy (30%)

Evaluates correct parameter usage with word-boundary matching in structured CALL blocks and code

Code Quality (35%)

Assesses working code production with correct endpoints and parameters in Python code blocks

Limitations

These preliminary results should be interpreted with the following constraints in mind:

  • Single-run data: n=1 per condition with no repetitions - statistical significance requires n≥3
  • Limited spec coverage: Only 7 of 50 planned specifications tested (14% coverage)
  • Format coverage: Only 2 of 5 formats tested (OpenAPI, AsyncAPI) - GraphQL, Postman, and Protobuf pending
  • Single model: Results only from Claude Sonnet 4.5 - multi-model comparison needed for generalization
  • No randomization: Tier order was fixed in execution - sequence effects not controlled
  • Task distribution: Uneven representation across API complexity levels

Next Steps

To reach statistically robust conclusions, the following work is planned:

  • Complete full benchmark: Execute all 500 runs (50 specs × 5 tiers × 2 tasks)
  • Add repetitions: Run each condition 3 times minimum for statistical significance testing
  • Multi-model evaluation: Test GPT-4, Gemini, and other leading models for cross-model validation
  • Randomize tier order: Eliminate potential sequence effects through randomized execution order
  • Statistical analysis: Conduct paired t-tests, ANOVA, and effect size calculations
  • Qualitative analysis: Manual review of failure cases to identify systematic patterns
  • Format-specific insights: Deep dive into GraphQL, Postman, and Protobuf results once available
